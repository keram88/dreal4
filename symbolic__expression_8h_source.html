101 /** Represents a symbolic form of an expression.
102 
103 Its syntax tree is as follows:
104 
105 @verbatim
106  E := Var | Constant(double) | RealConstant(double, double)
107  | E + ... + E | E * ... * E | E / E | log(E)
108  | abs(E) | exp(E) | sqrt(E) | pow(E, E) | sin(E) | cos(E) | tan(E)
109  | asin(E) | acos(E) | atan(E) | atan2(E, E) | sinh(E) | cosh(E) | tanh(E)
110  | min(E, E) | max(E, E) | if_then_else(F, E, E) | NaN
111  | uninterpreted_function(name, {v_1, ..., v_n})
112 @endverbatim
113 
114 In the implementation, Expression is a simple wrapper including a
115 pointer to ExpressionCell class which is a super-class of different
116 kinds of symbolic expressions (i.e. ExpressionAdd, ExpressionMul,
117 ExpressionLog, ExpressionSin).
118 
119 @note The sharing of sub-expressions is not yet implemented.
120 
121 @note -E is represented as -1 * E internally.
122 
123 @note A subtraction E1 - E2 is represented as E1 + (-1 * E2) internally.
124 
125 The following simple simplifications are implemented:
126 @verbatim
127  E + 0 -> E
128  0 + E -> E
129  E - 0 -> E
130  E - E -> 0
131  E * 1 -> E
132  1 * E -> E
133  E * 0 -> 0
134  0 * E -> 0
135  E / 1 -> E
136  E / E -> 1
137  pow(E, 0) -> 1
138  pow(E, 1) -> E
139  E * E -> E^2 (= pow(E, 2))
140  sqrt(E * E) -> |E| (= abs(E))
141  sqrt(E) * sqrt(E) -> E
142 @endverbatim
143 
144 Constant folding is implemented:
145 @verbatim
146  E(c1) + E(c2) -> E(c1 + c2) // c1, c2 are constants
147  E(c1) - E(c2) -> E(c1 - c2)
148  E(c1) * E(c2) -> E(c1 * c2)
149  E(c1) / E(c2) -> E(c1 / c2)
150  f(E(c)) -> E(f(c)) // c is a constant, f is a math function
151 @endverbatim
152 
153 For the math functions which are only defined over a restricted domain (namely,
154 log, sqrt, pow, asin, acos), we check the domain of argument(s), and throw
155 std::domain_error exception if a function is not well-defined for a given
156 argument(s).
157 
158 Relational operators over expressions (==, !=, <, >, <=, >=) return
159 symbolic::Formula instead of bool. Those operations are declared in
160 symbolic_formula.h file. To check structural equality between two expressions a
161 separate function, Expression::EqualTo, is provided.
162 
163 */

195  /** Checks structural equality.
196  *
197  * Two expressions e1 and e2 are structurally equal when they have the same
198  * internal AST(abstract-syntax tree) representation. Please note that we can
199  * have two computationally (or extensionally) equivalent expressions which
200  * are not structurally equal. For example, consider:
201  *
202  * e1 = 2 * (x + y)
203  * e2 = 2x + 2y
204  *
205  * Obviously, we know that e1 and e2 are evaluated to the same value for all
206  * assignments to x and y. However, e1 and e2 are not structurally equal by
207  * the definition. Note that e1 is a multiplication expression
208  * (is_multiplication(e1) is true) while e2 is an addition expression
209  * (is_addition(e2) is true).
210  *
211  * One main reason we use structural equality in EqualTo is due to
212  * Richardson's Theorem. It states that checking ∀x. E(x) = F(x) is
213  * undecidable when we allow sin, asin, log, exp in E and F. Read
214  * https://en.wikipedia.org/wiki/Richardson%27s_theorem for details.
215  *
216  * Note that for polynomial cases, you can use Expand method and check if two
217  * polynomial expressions p1 and p2 are computationally equal. To do so, you
218  * check the following:
219  *
220  * (p1.Expand() - p2.Expand()).EqualTo(0).
221  */

378  /** Constructs if-then-else expression.
379 
380  @verbatim
381  if_then_else(cond, expr_then, expr_else)
382  @endverbatim
383 
384  The value returned by the above if-then-else expression is @p expr_then if
385  @p cond is evaluated to true. Otherwise, it returns @p expr_else.
386 
387  The semantics is similar to the C++'s conditional expression constructed by
388  its ternary operator, @c ?:. However, there is a key difference between the
389  C++'s conditional expression and our @c if_then_else expression in a way the
390  arguments are evaluated during the construction.
391 
392  - In case of the C++'s conditional expression, <tt> cond ? expr_then :
393  expr_else</tt>, the then expression @c expr_then (respectively, the else
394  expression @c expr_else) is \b only evaluated when the conditional
395  expression @c cond is evaluated to \b true (respectively, when @c cond is
396  evaluated to \b false).
397 
398  - In case of the symbolic expression, <tt>if_then_else(cond, expr_then,
399  expr_else)</tt>, however, \b both arguments @c expr_then and @c expr_else
400  are evaluated first and then passed to the @c if_then_else function.
401 
402  @note This function returns an \b expression and it is different from the
403  C++'s if-then-else \b statement.
404 
405  @note While it is still possible to define <tt> min, max, abs</tt> math
406  functions using @c if_then_else expression, it is highly \b recommended to
407  use the provided native definitions for them because it allows solvers to
408  detect specific math functions and to have a room for special
409  optimizations.
410 
411  @note More information about the C++'s conditional expression and ternary
412  operator is available at
413  http://en.cppreference.com/w/cpp/language/operator_other#Conditional_operator.
414  */
